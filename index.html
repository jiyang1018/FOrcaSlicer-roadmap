Design decisions locked
  • wall[0] = 0.2mm outer nozzle; wall[1+] = 0.4mm inner nozzle
  • Min 2 walls when same color AND material; 1 wall allowed otherwise
  • Slope-compensated gap between wall[0] and wall[1] only
  • Accumulator sync: 0.4 nozzle fires when next 0.2 layer pushes accumulated height above 0.4 max
  • Color-to-nozzle priority: color match → smallest nozzle = outer → extruder 0 fallback
  • Color depth: min = nozzle diameter; thin section = bleed-through or 50/50 split
  • Per-patch nozzle override: global=YES follows mixed logic; global=NO single nozzle owns patch
  • Priority hierarchy locked: CL (color patch loops) > OW (outer wall) > IW (inner wall)
Decision change
  • Started on OrcaSlicer-FullSpectrum fork, later pivoted to Snapmaker Orca due to cleaner Filament for Features UI
  • FullSpectrum had resolved_model_config_for_tab stripping sparse_infill_filament — unfixable cleanly

Phase 5b
Full-wrap cylinder — root cause identified, pivot to new architecture
stolen pipeline fundamentally wrong for full-wrap — new color_patch_strip approach planned
▾
Problem discovered
  • When entire cylinder side painted red (CL=4, AW=2): expected 4 red loops + 2 white loops + white infill
  • Actual result: 1 red loop + 3 white loops, no infill — broken regardless of approach tried
  • Root cause: the original stolen pipeline extends the painted volume all the way to the model center — stolen = full cross-section for a fully painted face, not just the outer CL loops
  • mine = diff(layer_slice, segmented.expolygons) removes the full painted depth from white — white loses its AW loops too when painted volume > wall thickness
Key findings from investigation
  • CL=4 with 0.4mm nozzle: stolen is confirmed 1.6mm deep (4 × 0.4mm) — painting system does steal exactly CL loops deep ✅
  • But mine subtracts the full segmented.expolygons (full painted volume), not just the CL-loop strip — this is the bug
  • loop_number override in PerimeterGenerator asks red to print CL loops, but white still has geometry overlapping those positions because mine was computed wrong
  • color_patch_regions storing shell_strip = full stolen: inner_core = diff(stolen, stolen) = empty — nothing returned to white
  • Renaming "stolen" to "painted" noted as future cleanup (upstream variable, don't rename yet)
Failed approaches during Phase 5b — do not retry
  • ❌ Erode full_model by CL×loop_width to get boundary strip — collapses thin walls (AW=2 wall = 0.8mm, CL×0.4mm = 1.6mm > wall)
  • ❌ Erode stolen directly — produces holey annular ring, red prints outer+inner boundary only
  • ❌ Area threshold (stolen_area ≥ 85% full_area) to detect full-wrap — threshold unstable
  • ❌ Subtract color_patch_regions from base_dst in restore block — over-subtracts when shell_strip = full wall
  • ❌ wall_loops-based cap in PerimeterGenerator — wrong, CL is independent of AW
  • ❌ Multiple reverts and rebuilds — code state became inconsistent, required git checkout ba3d32bc68 multiple times
Architecture decision — new color_patch_strip pipeline
  • Leave stolen/mine pipeline completely untouched — it works correctly for all non-color-patch cases
  • New color_patch_mode bool added to PrintObjectConfig (per-object setting, default false)
  • color_patch_loops stays in PrintConfig (per-filament)
  • UI: MMU gizmo panel gets "Original" / "Color Patch" toggle at top; CL slider only shown in Color Patch mode
  • When color_patch_mode=true: new color_patch_strip computed as intersection(full_model_CL_boundary, painted_faces); replaces what stolen puts in by_region for painted extruder; mine subtracts color_patch_strip (not full segmented.expolygons) so white keeps its AW loops
  • Priority rule: CL > OW > IW — color patch loops are outermost regardless of AW setting

已锁定的设计决策
  • wall[0] = 0.2mm 外墙挤出口；wall[1+] = 0.4mm 内壁挤出口
  • 相同颜色且相同材料时，最少 2 条总墙壁；否则允许 1 条
  • 仅在 wall[0] 和 wall[1] 之间进行坡度补偿间隙
  • 累加器同步：当 0.2 层累计高度超过 0.4 挤出口最大层高时触发 0.4 挤出口
  • 颜色到挤出口优先级：颜色匹配 → 最小挤出口=外墙 → 挤出机 0 兜底
  • 颜色深度：最小值=挤出口直径；薄截面=渗透或 50/50 分割
  • 每区域挤出口覆盖：global=YES 遵循混合逻辑；global=NO 单挤出口拥有整个区域
  • 优先级层次已锁定：CL（着色区域循环数）> OW（外墙）> IW（内壁）

全包裹圆柱 — 根本原因已定位，转向新架构
stolen 流水线对全包裹场景根本错误 — 新的 color_patch_strip 方案待实现
▾
发现的问题
  • 整个圆柱侧面涂红（CL=4，AW=2）：期望 4 条红色循环 + 2 条白色循环 + 白色填充
  • 实际结果：1 条红色 + 3 条白色，无填充 — 所有尝试方案均失败
  • 根本原因：原始 stolen 流水线将涂色体积一直延伸到模型中心，stolen = 全截面（而非仅外层 CL 圈）
  • mine = diff(layer_slice, segmented.expolygons) 从白色中减去完整涂色深度 — 当涂色体积 > 壁厚时，白色的 AW 循环也被一并删除
调查中的关键发现
  • 确认：CL=4 时 stolen 深度为 1.6mm（4×0.4mm）——涂色系统确实精确窃取 CL 圈深度 ✅
  • 但 mine 减去的是完整 segmented.expolygons（全涂色体积），而非仅 CL 圈带 — 这是 bug 所在
  • PerimeterGenerator 中的 loop_number 覆盖让红色尝试打印 CL 圈，但白色几何仍与这些位置重叠（mine 计算错误所致）
  • "stolen" 改名为 "painted" 已记录为未来清理项（上游变量，暂不重命名）
架构决策 — 新的 color_patch_strip 流水线
  • completely 保留 stolen/mine 流水线不变 — 所有非着色区域场景均正确
  • 新增 color_patch_mode 布尔字段至 PrintObjectConfig（每对象设置，默认 false）
  • color_patch_loops 保留在 PrintConfig（每耗材设置）
  • UI：MMU 工具面板顶部新增"原始 / 着色区域"切换；CL 滑块仅在着色区域模式下显示
  • color_patch_mode=true 时：计算 color_patch_strip = intersection(模型CL边界带, 涂色面)；替换 stolen 在 by_region 中的内容；mine 减去 color_patch_strip（而非完整 segmented.expolygons），白色保留其 AW 循环
  • 优先级规则：CL > OW > IW — 着色区域循环始终位于最外层，与 AW 设置无关
